# Database Structure — Current Implementation

> **Audit note.** This file documents the database **as inferred from the
> migration files in `supabase/migrations/` and the TypeScript types in
> `lib/`**. Migrations are applied manually in Supabase, are idempotent, and
> each ends with `notify pgrst, 'reload schema';`. Because the live database is
> not introspected here, column lists are reconstructed from migrations + code.
> Anything not directly visible in a migration or a typed query is tagged
> **Needs confirmation**.
>
> Migration numbering note: **`040` does not exist** — the sequence jumps from
> `039_event_status_comments.sql` to `041_baseline_locked_at.sql`. There are
> also two non-numbered scripts: `RUN_THIS_FIRST_production_setup.sql` and
> `DIAGNOSTIC_workflow_state.sql`.

---

## 1. Engine & conventions

- **Postgres via Supabase**, with **Row Level Security (RLS)** enabled on
  effectively every business table.
- Auth identities live in `auth.users` (Supabase managed). Application tables
  reference `auth.users(id)` for ownership / authorship columns
  (`created_by`, `sales_owner_id`, `acknowledged_by`, …).
- Many derived numbers / sequences are generated by **SECURITY DEFINER RPCs**
  (`next_*_number`, `add_working_days`, admin/diagnostics functions).
- **Status fields are enforced with `CHECK` constraints**, not Postgres enum
  types (confirmed by the `check (… in (…))` clauses in migrations such as
  `008`, `017`, `067`, `063`).

## 2. Core business entities

The app is built around a **three-entity order lifecycle**:

```
documents (quotation)  ──►  production_task_lists  ──►  production_orders
        1                          1 (per won doc)              1 (per ready TL)
```

> POTENTIAL_INCONSISTENCIES.md).

**`production_orders.status` — 10-value CHECK** (`lib/types.ts ProductionOrderStatus`):

## 4. Identity, roles & access

### 4.1 `user_roles`

| Field | Notes | Source |
|---|---|---|
| `user_id` (FK → auth.users) | | base |
| `role` | CHECK: `admin`, `sales`, `task_list_manager`, `operations` | extended `030` (TLM), `042` (operations) |
| `super_admin` (boolean) | virtual elevated flag — **not** a role value | `016` |

- **`super_admin` is a boolean flag, never a DB `role` value.** Code surfaces it
  as a virtual role (`lib/auth.ts getCurrentUserRole`). RLS predicates check
  `coalesce(r.super_admin, false)` (seen in `067`). **Confirmed by code.**

### 4.2 `user_profiles` (m052)

- Display-name layer over `auth.users` (resolved by `lib/user-display.ts`).
  **Confirmed** (`052`).

### 4.3 Capability matrix (m026)

- `permissions` — catalog of capability keys.
- `role_permissions` — role × capability matrix.
- **App-level gating only** — the matrix governs `lib/permissions.ts`
  (`hasCapability`/`requireCapability`), NOT RLS. RLS is unchanged by the
  matrix. **Confirmed by code** (`lib/permissions.ts` comments; `026`, ops
  backfill `053`, factory-mapping cap `064`).
- The `lib/permissions.ts` `Capability` union has **22 keys** (see
  USER_ROLES.md §4). Whether the DB `permissions` catalog row count equals 22 is
  **Needs confirmation**.

### 4.4 Visibility scopes (m067) — `teams`, `team_members`, `access_grants`

- `teams` — `kind` CHECK `team` / `region` / `department`; self-referencing
  `parent_team_id`.
- `team_members` — `member_role` CHECK `member` / `manager`; PK (team_id, user_id).
- `access_grants` — `scope_type` CHECK `self` / `team` / `region` / `lens` /
  `all`; optional `lens_key` (`production`/`finance`/`logistics`),
  `expires_at` (delegation).
- **Additive migration**: no existing table's RLS was changed by `067`. The
  visibility engine (`lib/visibility.ts`) is **app-level only** with a legacy
  fallback (technical roles see all, sales see own) until grants are assigned.
  **DB-level RLS lens enforcement does NOT exist yet.** **Confirmed by code**
  (`067` header comment + `lib/visibility.ts`).

## 5. Operational / collaboration tables

| Table | Purpose | Source |
|---|---|---|
| `events` | Immutable (INSERT-only) polymorphic event log: `entity_type`, `entity_id`, `severity` (low/medium/high/critical), `status` (open/acknowledged/working/waiting/escalated/resolved) | `022`, status+comments `039` |
| `event_comments` | Threaded comments on events | `039`/`044` |
| `event_reads` | Per-user read state for comments | `045` |
| `entity_messages` | Polymorphic discussion/comment thread on document/task_list/production_order/client (m049). Only `message_kind='comment'` used today; `request`/`reply`/`structured_reply`/`parent_message_id`/`request_type` reserved for later phases | `049` |
| `quotation_reminders` | Sales follow-up reminders (`open`/`done`/`cancelled`), snooze presets | `043` |
| `attachments` | Affair-scoped files (`affair_id`, visibility flags, `attachment_type` CHECK) | `060` |
| `action_acks` | Action Center acknowledgements. PK `action_key`; `state` text default `'acknowledged'` (also `'done'`), **no CHECK constraint**; `acknowledged_by`, `acknowledged_at`, `note` | `069` |
| `action_notes` | Micro-notes pinned to action cards | `075` |

- **`action_acks.state` detail (Confirmed by code, `069`):** the `state` column
  is added both in the `create table` AND via a defensive
  `alter table … add column if not exists state …`. This matters for the
  "Done button" bug — see POTENTIAL_INCONSISTENCIES.md. RLS on `action_acks` is
  permissive (`using (true)`) for all authenticated users (read/insert/update/
  delete) — it is a shared operational marker.
- **`events` is append-only** — there is no UPDATE/DELETE path; status changes
  are modeled as new rows/comments (`lib/events.ts`). **Confirmed by code.**

                          production_ready | cancelled

                       in_production | production_delayed | production_completed |
                       shipment_booked | shipped | delivered | cancelled

Incoterm:              EXW | FOB | CFR | CIF | DDP | DDU
FreightType:           LCL | 20ft | 40ft HC
ContainerType:         LCL | 20ft | 40ft | 40ft HC   ← note: includes plain "40ft"

**`FreightType` vs `ContainerType` mismatch (Confirmed by code):**
`FreightType` is `LCL | 20ft | 40ft HC` (no plain `40ft`), while
`ContainerType` (and the `document_containers.container_type` CHECK fixed in
`063`) is `LCL | 20ft | 40ft | 40ft HC`. See POTENTIAL_INCONSISTENCIES.md.

## 8. Missing / unclear relationships

- **`shipping_details` jsonb has no DB-level schema.** Field shape is enforced
  only in `lib/shipping.ts` (`normalizeShippingDetails`). A second module
  (`lib/action-center.ts`) reads **different keys** (`forwarder_name`,
  `vessel_name`) than the ones written (`forwarder`, `vessel`). **Suspected
  inconsistency** — see SHIPPING_PROCESS.md & POTENTIAL_INCONSISTENCIES.md.
- **`bl_profile` jsonb has no DB-level schema** — shape enforced only in
  `lib/bl.ts` (`normalizeBlProfile`).
- **Visibility tables (`access_grants`) are not yet joined into RLS** — the
  link between a grant and what a user can query is enforced in app code only.
  **Confirmed** (`067`).
- **`actual_completion_date` exact column name Needs confirmation** (used by
  `components/dashboard/OrdersInFlight.tsx` via query alias).
- Exact column inventories for catalog tables (products, categories,
  factory_mappings, banks, sales terms) are **Needs confirmation** — not
  re-derived field-by-field in this audit.
