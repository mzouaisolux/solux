# DATABASE_MODEL.md — SOLUX Database Model Audit

> **Scope and method.** This document is produced by a read-only audit of
> `supabase/migrations/*.sql` (76 files, applied manually in Supabase, each
> ending with `notify pgrst, 'reload schema';`) and `lib/types.ts`. The live
> database was NOT introspected. Column lists are reconstructed from migrations
> and TypeScript types. Anything not directly proven by code is tagged
> **[Needs confirmation]**.
>
> **Tags used throughout:**
> - `[Confirmed by code]` — directly verified in cited migration or lib file.
> - `[Assumed from code]` — strongly implied by structure but not a single
>   explicit enforcement point.
> - `[Needs confirmation]` — inference or assertion that requires live DB
>   inspection or explicit owner input to verify.
>
> **Migration numbering note:** the sequence jumps from `039` to `041`. There
> is no `040_*.sql` file. This is confirmed and benign. [Confirmed by code]
>
> **Primary sources:** `docs/current-implementation/DATABASE_STRUCTURE.md`
> (already-audited; this document verifies and extends it); individual
> migrations cited as `mNNN`; `lib/types.ts` for enum definitions.

## 1. Engine and conventions

- **Postgres via Supabase**, RLS enabled on effectively every business table.
  [Confirmed by code]
- Auth identities live in `auth.users` (Supabase-managed). Application tables
  reference `auth.users(id)` for authorship columns (`created_by`,
  `sales_owner_id`, `acknowledged_by`, etc.). [Confirmed by code]
- **Status fields are enforced with `CHECK` constraints, not Postgres enum
  types.** Seen in `check (… in (…))` clauses in m008, m017, m018, m067, m063,
  etc. [Confirmed by code]
- Many derived numbers / sequences are generated by **SECURITY DEFINER RPCs**
  (`next_*_number`, `add_working_days`, admin/diagnostics functions).
  [Confirmed by code]
- All migrations are idempotent and end with
  `notify pgrst, 'reload schema';`. [Confirmed by code — every inspected
  migration]
- **Catalog column lists** (products, categories, factory_mappings, banks,
  sales conditions) are reconstructed from migration headings and TypeScript
  types. Exact column inventories for these tables are
  **[Needs confirmation]** — they were not re-derived field-by-field here.

### 2.3 Versioning model

Revision creates a **new row** sharing `root_document_id`; `version`
increments. Forecast / detail surfaces deduplicate to the latest version per
affair (confirmed by `lib/forecast.ts` + m059 comment). [Confirmed by code]

### 2.4 Advisory validation loop

`validation_status` is a parallel state to `status` (m068). It tracks
`none → pending → approved | rejected`. **This never blocks sending or winning
a quote** — explicitly stated in `lib/validation.ts` header. [Confirmed by
code]

### 4.5 Baseline and delay model

- **Production start date** = `coalesce(deposit_received_at, deposit_override_at::date)`. Not a stored column — derived on read. [Confirmed by code — m041 comment, `lib/production-lifecycle.ts`]
- **Baseline locks at production activation**, NOT at first working_days save. This was corrected in a May-2026 revision. [Confirmed by code — `lib/production-lifecycle.ts isBaselineLocked` comment]
- **Operational delay** = `current_production_deadline − initial_production_deadline`. [Confirmed by code — `lib/types.ts computeProductionDelay`, `lib/production-lifecycle.ts computeBaselineDelay`]

## 16. Unclear or inconsistent relationships

### 16.1 `production_status` / `task_list_status` are query aliases, not columns

**Current behavior:** `production_orders.status` is the real DB column.
Dashboard queries alias it as `status AS production_status`. TypeScript types
`OrderInFlight.production_status` and `OrderStageInput.production_status` use
the alias name. Similarly, `task_list_status` is `production_task_lists.status`
aliased.  
**Risk:** Any new query that selects `production_status` directly (without the
alias) would fail or return NULL.  
[Confirmed by code — `lib/lifecycle.ts`, `components/dashboard/OrdersInFlight.tsx`]

### 16.2 `shipping_details` jsonb has no DB schema; key mismatch bug

**Current behavior:** `production_orders.shipping_details` is an unstructured
jsonb column (m070). The shape documented in m070 is:
`{bl_number, forwarder, vessel, voyage, ...}`.  
`lib/shipping.ts ShippingDetails` writes **`forwarder`** and **`vessel`**.  
`lib/action-center.ts blIsFilled()` reads **`forwarder_name`** and
**`vessel_name`** — keys that are never written.  
**Effect:** entering only `forwarder` + `vessel` does not clear the "BL
missing" Action Center card. Only `bl_number` (or a filled client consignee
company name) actually clears it.  
[Confirmed by code — m070, `lib/shipping.ts`, `lib/action-center.ts`]

### 16.3 `action_acks.state` has no CHECK constraint

**Current behavior:** `state text not null default 'acknowledged'` with no
`CHECK (state in (...))` constraint. The only values the app writes are
`'acknowledged'` and `'done'`, but the DB does not enforce this.  
[Confirmed by code — m069]

### 16.4 `FreightType` vs `ContainerType` mismatch: plain `40ft`

**Current behavior:**  
- `FreightType` (`lib/types.ts`): `LCL | 20ft | 40ft HC` — **no plain `40ft`**  
- `ContainerType` (`lib/types.ts`) and `document_containers.container_type`
  CHECK (m063): `LCL | 20ft | 40ft | 40ft HC` — **includes plain `40ft`**

This means `documents.freight_type` cannot represent `40ft`, but
`document_containers.container_type` can. The vocabularies are intentionally
separate (freight type on the quote header vs container type in the container
plan) but the divergence may be unintentional.  
[Confirmed by code — `lib/types.ts`, m063]

### 16.5 Migration 040 is missing

The migration sequence jumps from `039_event_status_comments.sql` to
`041_baseline_locked_at.sql`. No file numbered `040` exists. This is confirmed
and benign (no migration was ever numbered 040).  
[Confirmed by code — `supabase/migrations/` listing]

### 16.6 `bl_profile` jsonb has no DB-level schema

**Current behavior:** `clients.bl_profile` is a free jsonb column (m054).
Shape is enforced only in `lib/bl.ts normalizeBlProfile`. There is no DB
CHECK, JSON schema validation, or trigger. Any direct SQL write can store an
arbitrary shape.  
[Confirmed by code — m054]

### 16.7 `actual_completion_date` exact column name

The `ProductionOrder` TypeScript type (`lib/types.ts`) declares
`actual_completion_date: string | null` and m018 creates the column as
`actual_completion_date date`. However, `lib/production-lifecycle.ts` uses
`actual_completion_date` to determine the `completed` lifecycle phase.
A separate concern: `lib/lifecycle.ts computeOrderFlightStage` uses PO status
`production_completed` (not this date column) for the "Production complete"
flight phase — these two completion signals can disagree. [Confirmed column
name — m018; **[Needs confirmation]** that they are never out of sync in
practice]

### 16.8 `production_working_days` vs `working_days` naming

The column on `production_orders` is `production_working_days` (m021, m041).
The TypeScript `ProductionOrder` type also shows `production_working_days`.
Some older migration comments or internal references may use `working_days`
(shorter form). The exact column name as currently in the live schema is
`production_working_days`. **[Needs confirmation]** via live `information_schema`
query if any residual references to a plain `working_days` column exist.

### 16.9 Capability `admin.diagnostics` may not have a DB row

`lib/permissions.ts Capability` union includes `admin.diagnostics` (22nd key).
Migrations m026, m053, m064 together seed 21 capability rows in the `permissions`
table. No migration seeds `admin.diagnostics`. The TypeScript union having this
key but the DB potentially lacking the row means `hasCapability('admin.diagnostics')`
always returns false unless the row was added manually.
**[Needs confirmation]** — query `select key from permissions where key = 'admin.diagnostics'`
in live DB.

### 16.10 Visibility grants not enforced at RLS level

`access_grants` (m067) defines `scope_type = 'lens' | 'team' | 'region' | 'all'`
but none of these are joined into any RLS policy. The link between a grant and
what a user can query is enforced **in app code only** (`lib/visibility.ts`).
A direct DB or RPC path that bypasses `lib/visibility.ts` can over-return rows.
[Confirmed by code — m067 header: "NO existing table's RLS is changed here"]
