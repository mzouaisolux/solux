## 2. Ce qui fonctionne bien

**Fonctionnalités solides (proches prod) :**
- **Cycle de validation Task List** (draft→under_validation→validated→production_ready) : états clairs, actions différenciées par rôle, garde-fous (mapping complet + pas de révision ouverte avant Release).
- **Résolution Factory mapping en couches** (global → client → order), auto-résolue sur la task list + dans le PDF usine. Élégant, réutilisable (`Copy mappings between families`).
- **Gate acompte** : 953,34 → auto-avancement `awaiting_deposit → deposit_received` → production déverrouillée + solde tracké. Impeccable en live.
- **Génération Commercial Invoice** : `CI-0002` = PDF export complet et correct (consignee, refs, FOB, lignes, total). Qualité « prêt à envoyer ».
- **Moteur de permissions** : matrice rôle×capacité, enforcement serveur (`requireCapability`), gating route (`admin/layout.tsx`), garde-fou anti-lockout super-admin. Vrai Sales/TLM correctement refusé (litmus vérifié).
- **Ledger de migrations** (m113) + sondes REST — a déjà rattrapé des migrations non appliquées en prod.

**Workflows fluides :** quote→task list→production order (Launch/Release en un clic), round-trip notifications Sales↔TLM, lifecycle commande par chips de statut.

**Particulièrement bien conçu :** la **séparation permissions (« que puis-je faire ») vs visibilité m067 (« que puis-je voir »)** — distinction que beaucoup d'apps ratent.

**Surprises positives :**
1. Profondeur du modèle production (attribution délai factory/external, baseline gelée, ETA).
2. Le garde-fou « désactiver manage_permissions pour super-admin te verrouille ».
3. La CI générée est *vraiment* complète, pas un placeholder.
4. Ledger migrations + sondes = maturité ops inattendue à ce stade.

## 4. Analyse métier Solux

Le workflow est **globalement cohérent** pour Solux (fabricant/exportateur d'éclairage solaire B2B projet). La chaîne *prospect → affaire → demande de service → devis → task list → validation production → commande → acompte → production → shipping/docs export* modélise bien un **export FOB Chine→Afrique**.

| Brique | Verdict |
|---|---|
| **CRM** (prospects/companies/tenders) | Pertinent (dédup name_key, attributions, AO). Beaucoup de surface — risque de sur-ingénierie vs adoption réelle. |
| **Projets / Affaires** | ✅ Hiérarchie Client→Affaire→Request (m124) résout la collision « projet ». |
| **Quotations** | ✅ Solide (pricing par liste, snapshots, n° immutable, affaire obligatoire). |
| **Requests (Service Requests)** | OK mais **3 services actifs / 6 futurs** — registre en avance sur l'usage. |
| **Task Lists** | ✅ Pièce maîtresse, bien pensée. |
| **Factory Mapping** | ✅ Très bon concept. **Mal protégé** : édition libre d'une config qui impacte tous les devis, sans diff/validation. |
| **Production** | Bon squelette, mais **`production_time` non capturé** casse le suivi de délai (à corriger côté devis). |
| **Communication interne** | Fonctionne (notifs 2 sens) mais **audit auteur faible** (ID brut). |
| **Validation des données** | Point faible transverse : pas de typage DB, dérive de config possible, peu de contraintes au compile. |

**Mal modélisé / à revoir :** (1) `production_time` doit devenir une donnée de premier ordre du devis ; (2) la matrice de permissions a besoin d'un **mode édition contrôlé** (diff + confirmation + alerte) ; (3) vérifier l'usage réel du CRM/Tenders avant d'y investir plus.

## 6. Dette technique

**Niveau global : MOYENNE-ÉLEVÉE** (élevée sur tests + typage ; moyenne ailleurs).

**Zones les plus fragiles :** `lib/supabase/*` (non typé, 10 erreurs tsc baseline) · listes `fetch-all + .limit(5000)` · `production/orders/actions.ts` (délai/ETA + dépend de `production_time` non rempli) · la matrice de permissions (donnée critique éditable sans garde-fou).

**À refactoriser :** générer les types Supabase (`<Database>`), extraire l'agrégation vers SQL/RPC, centraliser la pagination, typer les RPC.

**Si on ajoute des features sans refactoriser — dans 3 à 6 mois :**
- **Données** : listes tronquées → totaux/forecast faux → décisions sur de mauvais chiffres.
- **Régressions invisibles** : sans intégration, chaque feature peut casser le workflow (déjà arrivé : P0).
- **Bugs runtime en hausse** : plus de colonnes → plus de 42703 non attrapés au compile.
- **Sécurité** : plus de rôles/caps → plus de surface de dérive de la matrice.
- **Lenteur** : la falaise fetch-all se rapproche.

## 7. Risque de régression

- **À figer (+ tests de non-régression) :** cycle Task List (submit→validate→release), gate acompte, résolution Factory mapping, enforcement des capacités (Sales≠TLM≠Admin), n° de documents.
- **Change trop souvent :** CRM/prospects/tenders, dashboards (3 implémentations d'alertes), pricing — à stabiliser.
- **Tests automatiques prioritaires :** (1) enforcement permissions par rôle (rôle × route protégée), (2) transitions d'état task list/commande, (3) résolution mapping (override>client>global>missing), (4) calcul du gate acompte, (5) garde-fou matrice (super-admin ne peut pas se verrouiller).
- **Avant chaque release :** E2E Sales→TLM→Production en **vrais comptes** (pas Preview) + check sécurité (Sales refusé sur /permissions + /factory-mapping).

## 9. Avis personnel (honnête)

**Principales inquiétudes :**
1. **Le filet de sécurité est trop fin pour la richesse fonctionnelle.** L'app fait beaucoup (CRM, pricing, prod, export, permissions) sur 211 tests purs et zéro intégration. Château construit vite sur fondations de tests minces — le P0 et la faille matrice n'étaient détectables qu'en testant comme un vrai utilisateur.
2. **La couche données n'a pas de garde-fou compile** (Supabase non typé) — chaque colonne est un pari runtime.
3. **La config critique peut dériver en silence** (la matrice donnait des droits-dieu à un TLM en prod).
4. **Surface fonctionnelle > adoption prouvée** (tenders, prospects v3, i18n…) — risque de complexité non amortie.

**Ce qui me rassure :**
- Le **modèle métier est juste** — rare et précieux.
- Réflexes de maturité déjà là (ledger migrations + sondes, garde-fou anti-lockout, séparation permissions/visibilité, snapshots produit).
- Le propriétaire **teste sérieusement et accepte la critique** (cette mission E2E vrais comptes, l'insistance « pas de Preview »).

**Stratégie 30 jours — *« Geler, ceinturer, puis avancer. »***
- **Semaine 1** : geler les nouvelles features. tsc+lint en CI. Garde-fou matrice + alerte audit. Corriger `.limit(5000)`.
- **Semaines 2-3** : types Supabase, première vraie couche de tests (permissions/rôle + transitions), capturer `production_time`.
- **Semaine 4** : pagination des grosses listes, runbook « E2E vrais comptes avant release », puis rouvrir le développement.
